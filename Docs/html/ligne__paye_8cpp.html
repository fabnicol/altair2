Atteint le prochain noeud de libellé donné, après un saut éventuel, et le lit.

Paramètres
tagnom du noeud
curnoeud courant
lindice courant de la table
infotable d'informations
normalJumpnombre de noeuds sautés (défaut 0)

Va au prochain noeud de libellé tag, après un saut éventuel normalJump.
Assigne ce noeud XML dans le pointeur courant . Lit la propriété "V" de ce noeud dans la table info à l'indice l de l'agent courant.
Renvoie un code d'exception LINE_MEMORY_EXCEPTION en cas d'allocation mémoire impossible ou NO_NEXT_ITEM si drapeau_cont est vrai pour #info.
Sinon appelle sanitize et retourne NODE_FOUND

Renvoie
NODE_FOUND sauf si LINE_MEMORY_EXCEPTION ou NO_NEXT_ITEM.

static void GCC_INLINE sanitize ( xmlChar *  s,
const char  sep 
)
static

Remplace les occurrences d'un caractère à l'intérieur d'une chaîne xmlChar* par le
caractère '_'.


Windows Convertit l'encodage de la chaîne UTF-8 en Latin-1.
Autres Pas de conversion.

Paramètres
sChaîne à contrôler
sepCaractère à nettoyer (le séparateur des bases CSV)

Le caractère de remplacement ne doit jamais être séparateur de champ CSV.
Il est donc interdit d'avoir des bases de type CSV séparées par le caractère '_' (au lieu de ',' ou ';').

Attention
 Windows 

      Cette opération peut échouer si les hypothèses techniques suivantes, 
      relatives à la conversion Latin-1, ne sont pas remplies.
                           a) pas de caractères spéciaux multioctets
                           b) seuls sont convertis : à, â, ç, è, é, ê, ë, î, ï, ô, û ...
                              et les majuscules correspondantes autrement dit, dont le
                              code UTF-8 commence par 0xC3. Il suffit d'ajouter 0x40 sur
                              les quatre bits hauts de l'octet.
      Aucune vérification n'est opérée sur la réalisation de ces hypothèses. 
 Autres 

      La fonction ne convertit pas les caractères de sortie en Latin-1.
A faire:
Windows
Elaborer une vérification minimale des hypothèses.
Vérifier l'évolution du point suivant.
Le caractère '°' (degré) est bien codé en Latin-1 comme 0xB0, mais il y a un problème avec le paquet texlive
inputenc pour la conversion pdf. On remplace donc par (0x65). Apparemment plus nécessaire
1 if (info.Table[info.NCumAgentXml][l][i] == 0xB0)
2  info.Table[info.NCumAgentXml][l][i] = 0x65;
Note
A surveiller en cas de développement Windows.
